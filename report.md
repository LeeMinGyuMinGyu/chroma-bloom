## 1. 기획 (Tynan Sylvester, 『게임 기획의 정석』 프레임)

### 1.1 핵심 콘셉트
플레이어는 어두운 방에서 **단 하나의 동작 — "칠하기"** 만 가진다. 흰 표면에 색을 칠하면 그 색이
빛으로 번지고, 번진 빛이 길을 만들거나 장애물을 바꾼다. "빛을 직접 다루는 것"이 곧 퍼즐이다.

### 1.2 메카닉 → 이벤트 → 경험
- **메카닉:** 표면의 알베도(색)를 바꾸는 칠하기 + 그 색을 간접광으로 전파하는 GI.
- **이벤트:** 흰 패널을 빨강으로 칠하는 순간 빛이 번지며 *빛의 다리*가 나타난다.
- **경험:** "내가 칠한 색이 공간의 빛을 바꿨다"는 인과의 발견 — 통제감과 작은 쾌감.

![칠하기 전/후로 공간의 빛이 바뀌는 장면](captures/01_concept_before_after.png)
> ⛔ 캡쳐 삽입 필요 — 칠하기 전(흰 방) / 후(색이 번진 방) 비교

### 1.3 우아함(elegance) & 의미 있는 선택
"칠한다"는 단일 규칙이 색에 따라 다른 결과(다리·장애물·경로)를 낸다 → 적은 규칙으로 많은 상황.
어디에 무슨 색을 칠하느냐가 매번 의미 있는 선택이 되도록 설계했다.

| 🟡 노랑 | 노랑 바운스가 숨은 경로를 드러냄 → 이동 | 방3 확장 |

## 4. 강의 개념 ↔ 구현 매핑

> 강의 개념사전(L1~L6, L11, GI)을 본 게임 구현에 매핑. **각 행을 본인 게임 캡쳐로 설명.**

### 4.1 좌표계·변환 / 파이프라인·공간 (L1~L2)
- **Model(World) Transform:** 모든 오브젝트를 `mesh.position.set(...)`으로 OS→WS 배치(이동·스케일 어파인 조합).
- **View Transform / View Matrix:** 카메라가 WS를 카메라 공간으로 변환(1인칭 시점).
- **Perspective Projection / FOV / Frustum:** `PerspectiveCamera(72°, aspect, near 0.1, far 300)`.

![원근감 있는 방 + 오브젝트 배치](captures/05_transforms_projection.png)
> ⛔ 캡쳐 삽입 필요 — 원근감과 오브젝트 배치가 보이는 화면

### 4.2 래스터화·컬링 (L3)
- **Z-Buffer / Occlusion:** 가까운 면이 먼 면을 가리는 가시성(렌더러 자동).
- **AABB:** 플레이어 이동을 방 경계 상자로 클램프하고 틈 통과를 차단(`player.js`).

![벽/틈에서 막히는 장면(AABB 경계)](captures/06_aabb_block.png)
> ⛔ 캡쳐 삽입 필요 — 다리 없을 때 틈 앞에서 막히는 장면

### 4.3 라이팅·셰이딩 (L4)
- **Lₑ (방출 복사휘도):** 칠한 패널과 다리 타일의 `emissive` 발광.
- **Direct vs Indirect Lighting:** 직접광(키 라이트) + 간접광(DDGI 번짐). 본 게임의 핵심 대비.
- **Ambient / Diffuse / Attenuation:** Hemisphere·Ambient(주변광), 람베르트 난반사, 거리 감쇠.
- **Per-Pixel(Phong) Shading:** 노드 머티리얼의 프래그먼트 단위 조명.

![발광하는 패널/타일(Lₑ)](captures/07_emissive.png)
> ⛔ 캡쳐 삽입 필요 — 빛나는 패널 또는 다리 타일 클로즈업

### 4.4 텍스처·맵 (L5~L6) — *대비 서술*
- **Albedo:** 칠하기 = 표면의 순수 재질색(알베도) 변경. 조명은 GI로 별도 적용.
- **Light Map / Baking과의 대비:** 정적 라이트맵을 굽지 않고 **동적 GI**로 간접광을 즉시 계산 → 칠한 색이
  실시간으로 번질 수 있는 이유.

![칠하기로 알베도가 바뀌는 표면](captures/08_albedo.png)
> ⛔ 캡쳐 삽입 필요 — 흰(알베도 높음) 패널이 칠해지는 전/후

### 4.5 회전 표현 (L11)
- **Euler Angles(Yaw·Pitch):** 시점을 `Euler(pitch, yaw, 0, 'YXZ')`로 구성(`player.js`).
- **Quaternion:** `camera.quaternion.setFromEuler(...)`로 자세 적용.
- **Gimbal Lock 회피:** pitch를 ±90° 직전으로 클램프.

![상하·좌우 시점 회전](captures/09_rotation.png)
> ⛔ 캡쳐 삽입 필요 — 위/아래·좌/우를 둘러본 시점 두 장

> *(선택 확장)* 1인칭↔탑뷰 전환을 **SLERP**로 구현하면 L11 방향 보간을 추가로 시연 가능.

## 5. GI 구현 상세 — DDGI (L11 Global Illumination)

> 본 게임의 GI는 **DDGI(Dynamic Diffuse Global Illumination)** 를 적용한다.
> 강의 개념사전의 용어(프로브 / SH / 8-probe trilinear / spherical Fibonacci / 코사인 가중 반구 샘플링 /
> recursive feedback)를 그대로 구현·서술한다.

### 5.1 알고리즘
1. **Probe:** 월드 공간 3D 격자에 프로브를 배치(`gi/ddgi.js`의 프로브 그리드).
2. **광선 캡처:** 각 프로브에서 **spherical Fibonacci** 분포 + **cosine-weighted hemisphere** 표본화로
   광선을 쏴 교차점의 (직접광 + 알베도) 복사휘도를 모은다.
3. **저장(SH):** 프로브별 복사휘도를 **Spherical Harmonics 계수**로 저장.
4. **재사용/무한 바운스:** 저장된 프로브 조명을 다시 광선에 사용(**recursive feedback**)해 프레임에 걸쳐 누적.
5. **셰이딩:** 표면 점에서 **8-Probe Trilinear Interpolation** 으로 간접 irradiance를 구해 알베도와 곱해 더한다.

![GI On/Off 비교(간접광 유무)](captures/10_gi_onoff.png)
> ⛔ 캡쳐 삽입 필요 — GI 끈 화면 vs 켠 화면(색 번짐 차이)

![프로브 그리드 디버그 시각화](captures/11_probe_debug.png)
> ⛔ 캡쳐 삽입 필요 — `ddgi.js`의 프로브 디버그 뷰(점 격자)

![칠한 색이 옆 면으로 번지는 color bleeding](captures/12_color_bleeding.png)
> ⛔ 캡쳐 삽입 필요 — 빨강 패널 주변 흰 벽이 붉게 물든 장면

### 5.2 본 게임에서 GI가 필연인 이유
퍼즐의 단서는 "어떤 색이 *번졌는가*"로만 읽힌다. 직접광/플랫 라이팅에서는 칠한 면만 색이 보이고
주변으로 번지지 않아 단서가 화면에 나타나지 않는다 → **간접광(GI)이 퍼즐을 가시화하는 유일한 수단.**

### 5.3 한계 (정직한 서술)
- **Light Leak:** 프로브 격자 간격보다 작은 구조에서 조명 누수 가능(DDGI의 알려진 한계).
- 디퓨즈 간접광에 한정(정반사는 별도 기법 필요).

> **DDGI 구현 진행도에 따라:** 만약 제출 시점에 DDGI가 미완성이고 임시 바운스만 있다면,
> 이 절은 *실제 구현한 내용 그대로* 서술하고 DDGI는 "향후 작업"으로 명시한다(허위 기재 금지).
